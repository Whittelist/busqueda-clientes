## 3. Logica de la Campana

### 3.1 Reglas
- **Maximo 25 emails/dia** (sumando todos los toques)
- Solo **L-V** (fines de semana saltan)
- **Espaciado graduado** entre toques: 4, 6, 7, 7 dias (Dia 1, 5, 11, 18, 25)
- Los **follow-ups (touch 2-5) tienen prioridad** sobre nuevos envios para mantener cadencia
- Si un follow-up cae en finde → se desplaza al lunes

### 3.2 Estados y transiciones

```
pendiente  → email_1 (touch 1)  [hoy]
email_1    → email_2 (touch 2)  [hoy + 5 dias]
email_2    → email_3 (touch 3)  [hoy + 11 dias]
email_3    → email_4 (touch 4)  [hoy + 18 dias]
email_4    → email_5 (touch 5)  [hoy + 25 dias]
email_5    → fin (no mas)

CUALQUIER ← respondido (reply detectado en inbox)
CUALQUIER ← rebotado (bounce detectado)
CUALQUIER ← descartado (manual humano)
```

### 3.3 Algoritmo diario (campaign_runner.py)

```
PASO 1 - Detectar respuestas:
   Query Gmail: "to:equipo@silvigon.com -from:equipo@silvigon.com is:unread"
   Por cada reply:
     - Extraer From email
     - Buscar empresa en Sheet por columna E (email)
     - Marcar Estado = "respondido" + Fecha respuesta = hoy
     - Marcar en BD local (thread_id) para no re-procesar

PASO 2 - Leer Sheet:
   Todas las filas de "TEST EMPRESAS BOT 2"
   Mapear headers por nombre (alias detection como en main.py existente)

PASO 3 - Clasificar candidatos:
   - Tienen email valido (col D no vacio, no "No encontrado")
   - Estado NO es "respondido"/"rebotado"/"descartado"
   - pendiente → necesita touch 1
   - email_1 AND Ultimo contacto > 7 dias → necesita touch 2
   - email_2 AND Ultimo contacto > 7 dias → necesita touch 3

PASO 4 - Priorizar:
   1. Touch 3 pendientes (mas atrasados)
   2. Touch 2 pendientes
   3. Touch 1 nuevos (priorizar los que tienen problema_detectado relleno)

PASO 5 - Enviar 25 maximo:
   Para cada empresa (hasta 25):
     a. Generar email con DeepSeek (reusa prompts de main.py)
     b. Enviar DIRECTO via Gmail API (no draft)
     c. Actualizar Sheet: Estado, Ultimo contacto, Proxima accion, Numero Contactos
     d. Registrar en BD local (tabla envios)
     e. Sleep 2-5s entre envios

PASO 6 - Log y resumen
```

## 4. Estrategia de Contenido — Los 3 Emails (Diseñados desde Cero)

### Investigacion 2026 aplicada a nuestra campana

Basado en el analisis de 12M+ de cold emails en 2026:
- **3 emails capturan ~12% de respuestas** (4-5 emails llegan a 18%, pero 3 es un buen inicio)
- **Cada email debe anadir valor NUEVO** — nada de "just checking in"
- **Texto plano, max 80-125 palabras** — los compradores de construccion leen emails en 5 segundos desde el telefono en obra
- **Martes-jueves 7-11am** = mejor ventana de envio
- **Siempre en la misma conversacion** (Re: mismo asunto) — mejora entregabilidad y parece natural
- **Email 3 tipo "breakup"** genera MAS respuestas que un intento agresivo
- **25/dia desde 1 buzón** es seguro para un dominio con reputacion

## 5. Gmail API — Cambios Necesarios

### Token OAuth actual (`Automatizar emails 2/google_token_sheets.json`)
```json
"scopes": [
  "https://www.googleapis.com/auth/gmail.compose",
  "https://www.googleapis.com/auth/spreadsheets"
]
```

- `gmail.compose` permite **crear borradores y enviar mensajes** ✅ OK para enviar
- **Falta `gmail.readonly`** para detectar respuestas en el inbox
- Scopes necesarios para el nuevo token:
  ```json
  [
    "https://www.googleapis.com/auth/gmail.send",
    "https://www.googleapis.com/auth/gmail.readonly",
    "https://www.googleapis.com/auth/spreadsheets"
  ]
  ```

**Accion:** Re-autorizar una vez con `--auth` flag, igual que en main.py existente.

### Endpoints usados
| Operacion | Endpoint Gmail |
|-----------|---------------|
| Enviar email directo | `users.messages.send` (raw RFC 2822) |
| Detectar replies | `users.messages.list` query: `to:equipo@silvigon.com -from:equipo@silvigon.com is:unread` |
| Detectar rebotes | `users.messages.get` check headers for bounce/Delivery Status |
